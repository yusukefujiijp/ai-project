# Thread-Handoff Builder

## 0. Purpose

`_thread-end/thread-handoff.md` は、Thread移行時に `handoff.md` を作るためのBuilder Manualである。

このfile自体は `handoff.md` ではない。

```text id="core-distinction"
thread-handoff.md:
  Builder Manual.

handoff.md:
  Generated Product.
  Next-thread ignition key.
```

このfileの目的は、次Threadを安全に再起動するために、何を渡すべきかを最小・明確・再現可能な形で決めることである。

CPLM上、このFileは **Builder-light** として扱う。  
つまり、FrontmatterとCurrent CoordinateでBuilderの盤面を起動し、Programming-Like BlockでProduct境界・素材選別・圧縮・順序・Stop Conditionsを固定する。  
ただし、`handoff.md` product本文そのものをこのFileに固定しない。

このFileの画竜点睛は、Handoffを「要約」ではなく **Ignition Key Builder** として扱う点にある。  
`handoff.md` は、すべてを載せる箱ではない。  
次Threadが最初の一手を誤らないための鍵である。

### Programming-Like Block

```yaml id="purpose-builder-light"
thread_handoff_builder:
  role:
    - "Handoff Builder"
    - "Next-thread Ignition Manual"
    - "Builder-light CPLM"
    - "Dragon-Eye Product Boundary Vessel"

  not_role:
    - "handoff.md product itself"
    - "Harvest archive"
    - "thread-harvest.md duplicate"
    - "GitHub auto-commit executor"

  product:
    file: "handoff.md"
    role:
      - "Generated Product"
      - "Next-thread ignition key"

  core_distinction:
    - "thread-handoff.md builds."
    - "handoff.md ignites."
    - "Harvest gives material."
    - "Handoff compresses and orders."

  dragon_eye:
    - "Make Builder/Product boundary impossible to miss."
    - "Make handoff.md small enough to start."
    - "Make handoff.md strong enough to preserve context."

  guard:
    - "Do not confuse Builder with Product."
    - "Do not turn handoff.md into full Harvest archive."
    - "Do not turn thread-handoff.md into thread-harvest.md duplicate."
```

## 0B. Current Coordinate / 現在座標

このFileの現在座標は、Thread-End familyにおける **Handoff Builder** である。

`thread-handoff.md` は、`handoff.md` を作るためのBuilder Manualである。  
しかし、このFile自体は `handoff.md` ではない。

CPLM的には、FrontmatterがBuilderとしての正準Identityを起動し、このCurrent Coordinateが現在盤面を立ち上げる。

```text id="builder-coordinate"

  canonical Builder identity

Current Coordinate:
  current Handoff board
```

現在の最重要同期点は、`thread-end.md` / `thread-harvest.md` で確定したDownload Runtimeである。

```text id="handoff-runtime-coordinate"

  Harvest Download
  Read Harvest
  Handoff Download
```

したがって、HandoffはHarvestより先に作らない。  
Harvestが存在する場合、Handoffは **Read Harvest後** に作る。

## 0C. Handoff DNA Double Helix / 引き継ぎ二重螺旋

Handoffには二つの層がある。

一つは、人間の意味層である。  
なぜこのThreadを閉じ、何を次Threadへ渡すのか。  
何がRootで、何がFruitか。  
何を勝手に続けてはいけないか。

もう一つは、AIの実行層である。  
どのfileを読み、何を抽出し、どの密度で `handoff.md` を作り、どこで止まるか。

この二層が絡む時、`handoff.md` は単なる要約ではなく、次ThreadのIgnition Keyになる。

```text id="handoff-dna-compression"
Human Meaning Layer:
  why / root / context / caution / next intention

AI Execution Layer:
  read order / extraction / density / product boundary / stop condition
```

`thread-handoff.md` は、この二層を結ぶBuilderである。  
しかし、Productそのものではない。

### Programming-Like Block

```yaml id="handoff-dna-double-helix"
handoff_dna_double_helix:
  human_meaning_layer:
    preserves:
      - "Why the Thread is being transferred."
      - "What meaning must not be lost."
      - "What the user has sealed or not sealed."
      - "What is Root and what is Fruit."
      - "What must not be done next."

  ai_execution_layer:
    fixes:
      - "Priority Read Order."
      - "Harvest read status."
      - "Material extraction after Read Harvest."
      - "Density mode: minimal / standard / deep."
      - "One Next First Task."
      - "Stop after Handoff Download."

  builder_role:
    thread_handoff_md:
      role: "Bridge between meaning and execution"

        - "Select."
        - "Compress."
        - "Order."
        - "Seal."
      does_not:
        - "Replace Harvest."
        - "Become handoff.md."
        - "Continue after Stop."

  product_role:
    handoff_md:
      role: "Next-thread ignition key"
      must_be:
        - "Readable first."
        - "Small enough to start."
        - "Strong enough to preserve context."
        - "Clear enough to prevent wrong next action."

  compression:
    - "Meaning gives soul."
    - "Execution gives rails."
    - "Handoff gives ignition."
```

## 2. Builder / Product Distinction

`thread-handoff.md` はBuilder Manualである。  
`handoff.md` はGenerated Productである。

この区別が崩れると、Future AIはBuilderをProductとして扱い、次Thread起動用の軽い鍵を失う。

### Programming-Like Block

```yaml id="builder-product-distinction"
thread_handoff_md:
  type: "builder_manual"
  purpose:
    - "handoff.mdの作成手順を定義する"
    - "次Threadへ渡す情報を選別する"
    - "過剰なHarvest全文コピーを防ぐ"
    - "Priority Read Order / Current State / Next First Taskを固定する"
    - "Download RuntimeでHandoff Downloadを作る条件を固定する"

handoff_md:
  type: "generated_product"
  purpose:
    - "次Threadを再起動する"
    - "Future AIに最初に読ませる"
    - "現在地・未完了・次の一手を伝える"
    - "必要なread orderを示す"
    - "Harvest全体ではなく、起動に必要な鍵を渡す"

guard:
  - "Do not confuse thread-handoff.md with handoff.md."
  - "Do not turn handoff.md into full Harvest archive."
  - "Do not turn thread-handoff.md into thread-harvest.md duplicate."
  - "Do not generate handoff.md before Harvest is read when Harvest exists."
```

## 4. Handoff Density Mode

`handoff.md` は、Thread状況に応じて密度を選ぶ。

Defaultは `standard` とする。  
ただし、Next First Taskが明確で、変更fileが少なく、再起動経路が明白な場合は `minimal` でもよい。  
大きなProtocol transitionやMigration riskがある場合は `deep` を使ってよい。

重要なのは、`deep` でもfull Harvest copyにしないことである。

### Programming-Like Block

```yaml id="handoff-density-mode"
handoff_density_mode:
  minimal:
    use_when:
      - "next task is simple"
      - "few files changed"
      - "continuation path is obvious"
    include:
      - "Priority Read Order"
      - "Current State"
      - "Next First Task"
      - "Not-Now"
      - "Root Guard"

  standard:
    use_when:
      - "normal thread transfer"
      - "multiple files or decisions exist"
      - "Download Runtime follows meaningful Thread-End"
    include:
      - "Priority Read Order"
      - "Current State"
      - "Completed Updates"
      - "Unresolved Not-Now"
      - "Next First Task"
      - "Reality Response Needed"
      - "Root / Fruit Guard"
      - "Stop Conditions"

  deep:
    use_when:
      - "large protocol transition"
      - "Mission / Harvest / Migration risk exists"
      - "Future AI may lose context"
      - "multiple committed files require careful restart"
    include:
      - "standard fields"
      - "decision genealogy"
      - "risk map"
      - "seed routing"
      - "rollback conditions"

download_runtime_density:
  default_density: "standard"
  reason:
    - "Download Runtime usually follows a meaningful Thread-End."
    - "Next Thread needs enough restart context."
    - "Handoff must still stay lighter than Harvest."

  minimal_allowed_if:
    - "Thread transfer is simple"
    - "Few files changed"
    - "handoff.md can still define Priority Read Order / Current State / Next First Task"

  deep_allowed_if:
    - "large protocol transition"
    - "Future AI may lose critical context"
    - "multiple committed files require careful restart"

  guard:
    - "Deep does not mean full Harvest copy."
    - "Minimal still requires Priority Read Order and Next First Task."
    - "Standard is default for Handoff Download unless user specifies otherwise."

default_density: "standard"
```

```yaml id="priority-read-order-builder"
    - "latest updated project files"
    - "task/todo/lesson files if directly relevant"

    - "too many old files"
    - "obsolete versions unless genealogy matters"

priority_read_order_sync_guard:

  - "Do not start next_thread by reading full Harvest first unless handoff.md points there."
```

## 11. Next First Task Builder

### Programming-Like Block

```yaml id="next-first-task-builder"
next_first_task_rule:
  one_primary_task: true

  must_be:
    - "executable in the next Thread"
```
