## 2. Source Boundary / Tool Reality Guard

Safety Checkの内部理由をAIは直接見られない。

したがって、次を分ける。

```yaml id="tool-reality"
confirmed:
  - "tool call was blocked"
  - "commit did not happen"
  - "a smaller payload may or may not pass"
  - "manual commit may still be possible"

inference:
  - "payload length may be involved"
  - "protocol density may be involved"
  - "semantic mix may be involved"
  - "commit message / path / prior context may be involved"

not_confirmed:
  - "exact OpenAI internal safety reason"
  - "content is bad"
  - "root guard caused the block"
  - "placeholder body is bad"
```

原則：

```text id="boundary-rule"
Do not claim the exact internal safety reason.
Do not treat inference as confirmed.
```

日本語：

```text id="boundary-rule-ja"
内部理由を断定しない。
推測を確定事実として扱わない。
```

## 3. Transport-before-Semantics

Safety Block後の第一仮説は、内容の悪さではなく、搬送条件とする。

```text id="transport-before-semantics"
Transport before semantics.
Payload before content rewrite.
```

診断順序：

```yaml id="diagnosis-order"
diagnosis_order:
  1: "payload length / 本文量"
  2: "section density / セクション密度"
  3: "protocol density / Rule・Guard密度"
  4: "command style / 命令文構造"
  5: "semantic mix / 語彙組み合わせ"
  6: "commit message / path / prior context"
  7: "specific terms / 個別語彙"
```

内容改変は、搬送量・密度・分割単位を見た後に検討する。

## 6. Staged Canonicalization

長い本文が一度にCommitできない場合、意味を削るのではなく段階Commitへ分ける。

```text id="staged-canonicalization"
Commit the address first.
Grow the body by Git history.
```

日本語：

```text id="staged-canonicalization-ja"
まず住所をCommitする。
本文はGit履歴で育てる。
```

推奨Pattern：

```yaml id="staged-pattern"
staged_canonicalization:
  stage_1:
    name: "minimal canonical index"
    purpose: "stable path and source_bootstrap"

  stage_2:
    name: "core expansion"
    purpose: "core formula and role boundary"

  stage_3:
    name: "guard expansion"
    purpose: "misread guard and recovery guard"

  stage_4:
    name: "field expansion"
    purpose: "examples and detailed operating notes"
```

ただし、Human Direct Commitで全文貼り替えが可能な場合は、段階CommitよりFull Body Replacementが速い場合がある。

## 6.5 Try Once Then Fallback

Blocked write is not fully predictable in advance.

Therefore, if the original intended content is public-safe and Human Seal has been received, AI may try the original write once.

If it passes, the GitHub commit path is complete.

If it is blocked, stop immediately.

Do not repeatedly retry the same blocked payload.

Do not degrade the original intended content merely to satisfy tool transport.

```text id="try-once"
Try original once.
If blocked, preserve meaning.
Change the transport.
```

日本語：

```text id="try-once-ja"
本来版で一回は試す。
ブロックされたら意味を守る。
搬送経路を変える。
```

This is not a safety bypass.

This is a transport recovery route under Human Seal.

## 6.6 Human Direct Commit Fallback

When AI tool write is blocked, the workflow itself is not blocked.

```text id="tool-block"
Tool block is not workflow block.
Manual commit is not failure.
It is Reality Response.
```

AI should switch roles.

```yaml id="human-direct-roles"
ai_role_after_block:
  - "report the block honestly"
  - "do not claim the exact internal safety reason"
  - "do not conclude that the content is bad"
  - "preserve the original intended content"
  - "prepare a Manual Commit Pack"
  - "provide commit message"
  - "provide extended description"
  - "provide verification checklist"

human_role_after_block:
  - "open GitHub UI"
  - "paste or replace the target file body"
  - "commit directly"
  - "return Reality Response"
```

日本語：

```text id="human-direct-ja"
AI tool writeが止まっても、Workflowは止まらない。
AIは本来版を守り、Manual Commit Packを作る。
人間がGitHub UIでDirect Commitする。
AIはcommit後にVerificationする。
```

## 6.7 Address Placeholder Rail

If the target file does not exist and the original body is blocked, AI may create a minimal placeholder after Human Seal.

The placeholder is not the real content.

It is only a path anchor.

```markdown id="placeholder-classic"
# Pending Human Direct Commit

This file is a temporary path anchor.
Replace this body with the original intended version.
```

```text id="placeholder-classic-core"
Placeholder creates the address.
Human replaces the body.
Git history grows the file.
```

日本語：

```text id="placeholder-classic-ja"
仮ファイルは本体ではない。
仮ファイルは住所である。
人間が本来版へ全文置換する。
```

Use this only when it reduces human friction.

If the target file already exists, do not create a placeholder.  
Use the existing file as the edit target.

## 6.7.5 超中立Placeholder Rail / Ultra-Neutral Placeholder Rail

AI tool writeで本来版がblockされた時、target pathを先に開く必要がある場合、AIは **超中立placeholder** を一回だけ試してよい。

超中立placeholderは、本文ではない。

safe版ではない。

弱めた版ではない。

妥協版ではない。

それは、GitHub上の住所を開けるための最小鍵である。

```text id="ultra-neutral-placeholder-core"
placeholderは本文ではない。
placeholderは住所である。
```

English anchor:

```text id="ultra-neutral-placeholder-anchor"
Ultra-Neutral Placeholder = address opener, not content.
```

最初に試すplaceholderは、意味をほぼ持たせない。

Commit message:

```text id="ultra-neutral-placeholder-commit"
docs: add placeholder file
```

Content:

```text id="ultra-neutral-placeholder-content"
TODO
```

目的：

```yaml id="ultra-neutral-placeholder-purpose"
purpose:
  - "target pathを作る"
  - "GitHub edit surfaceを開く"
  - "人間が本来版を貼れる状態にする"
  - "AI側のsemantic payload搬送を止める"
  - "Human Direct Commitへ滑らかに渡す"
```

避けること：

```yaml id="ultra-neutral-placeholder-avoid"
avoid:
  - "placeholder内に長い説明を書く"
  - "Human Direct Commitなどの意味語をplaceholder本文へ入れる"
  - "original intended contentなどの説明をplaceholder本文へ入れる"
  - "replace this bodyなどの指示文をplaceholder本文へ入れる"
```

理由：

```text id="ultra-neutral-placeholder-reason"
placeholderの仕事は説明ではない。
住所を開けることである。
```

## 6.7.6 住所作成後深追い禁止Rule / Stop-after-Anchor Rule

超中立placeholderが通ったら、AI側で本体payloadをさらに通そうと深追いしない。

placeholder成功とは、AIが続きを頑張る合図ではない。

placeholder成功とは、GitHub上の住所が開き、人間が本来版を貼れる状態になったという合図である。

```text id="stop-after-anchor-core"
住所が開いたら、AI側搬送を止める。
Human Direct Commitへ切り替える。
```

English anchor:

```text id="stop-after-anchor-anchor"
Stop-after-Anchor = stop AI-side transport after path creation.
```

placeholder成功が意味すること：

```yaml id="placeholder-success-means"
success_means:
  - "target pathが存在する"
  - "GitHub UIでEditできる"
  - "人間が本来版を全文貼り替えできる"
  - "AIはManual Commit Packを提示する"
```

placeholder成功が意味しないこと：

```yaml id="placeholder-success-does-not-mean"
success_does_not_mean:
  - "AI側でさらにpayload実験を続ける"
  - "本来版を分割して通そうと深追いする"
  - "Root / Fruit Guardを弱める"
  - "safe-softened versionへ劣化させる"
  - "tool通過ゲームに時間を溶かす"
```

一文圧縮：

```text id="stop-after-anchor-final"
placeholderで住所が開いたら、AIは深追いせず、人間に本体を渡す。
```

## 6.7.7 Placeholder Block診断Guard / Placeholder Block Diagnosis Guard

超中立placeholderまでblockされた場合でも、placeholder本文が悪いとは断定しない。

`TODO` が悪いと判断してはいけない。

その場合は、tool action / target path / commit message / 直前block文脈 / payload構成 / internal safety state などの層を分けて考える。

```yaml id="placeholder-block-diagnosis"
possible_block_layers:
  - "tool action"
  - "target path"
  - "commit message"
  - "previous blocked context"
  - "payload structure"
  - "internal safety state"
```

してはいけないこと：

```yaml id="placeholder-block-do-not"
do_not:
  - "exact internal safety reasonを断定する"
  - "TODOが悪い内容だと診断する"
  - "本来版を弱める"
  - "Root / Fruit Guardを削る"
  - "同じpayloadを何度もretryする"
  - "placeholderをsemantic workaround化する"
```

正しい反応：

```text id="placeholder-block-response"
TODO placeholderまで止まるなら、
AI側retryを止める。
Download付き本来版を出す。
人間がGitHub UIで作成する。
Commit後にAIが検証する。
```

English anchor:

```text id="placeholder-block-anchor"
Placeholder block is not content judgment.
```

日本語圧縮：

```text id="placeholder-block-final"
placeholderが止まっても、本文が悪いとは限らない。
AIは断定せず、Human Direct Commitへ渡す。
```

## 6.8 Copy & Paste Pack Rail

If AI cannot safely transport the full content through GitHub tool write, AI should provide a Copy & Paste Pack.

```yaml id="copy-paste-pack"
copy_paste_pack:
  includes:
    - "target path"
    - "original intended Markdown body"
    - "downloadable Markdown file if useful"
    - "ZIP bundle if multiple files are involved"
    - "commit message"
    - "extended description"
    - "manual GitHub UI steps"
    - "post-commit verification checklist"
```

Purpose:

```text id="copy-paste-purpose"
Preserve original meaning.
Reduce human friction.
Move final execution to the human owner.
Verify repository reality after commit.
```

This route should be used when tool passage would require weakening the original intended content.

```text id="preserve-fire"
Do not degrade meaning.
Change transport.
Preserve the fire.
```

## 6.8.5 全文置換Manual Commit Pack Rule / Full-Body Manual Commit Pack Rule

Patch Previewは、判断のために使う。

Download付きManual Commit Packは、実行のために使う。

したがって、Manual Commit PackとしてDownload fileを出す場合、原則としてPatch-onlyではなく、target file全体を置換できる **Full Body Replacement File** を出す。

```text id="full-body-core"
Patchは判断用。
全文は実行用。
```

English anchor:

```text id="full-body-anchor"
Patch is for judgment.
Full body is for action.
```

### なぜFull Bodyをdefaultにするのか

人間がGitHub UIでDirect Commitする時、最短手順は次である。

```text id="human-direct-fastest"
対象fileを開く。
Editする。
既存本文を全選択する。
全削除する。
Full Bodyを全文貼り替える。
Commit messageとExtended descriptionを貼る。
Commitする。
```

この流れは、Patch挿入より速い。

理由：

```yaml id="full-body-reason"
reason:
  - "挿入位置を探さなくてよい"
  - "patch前後のmerge mistakeを減らせる"
  - "Markdown fence崩れを減らせる"
  - "全文置換なのでVerificationが単純になる"
  - "人間側の認知負荷が低い"
  - "AI toolがblockされた後の実行面では最短最速になりやすい"
```

### Patch-only Downloadの扱い

Patch-only Downloadは、Preview補助としては有効である。

しかし、Manual Commit Packのdefaultではない。

```yaml id="patch-only-position"
patch_only_download:
  valid_as:
    - "Patch Preview"
    - "Review aid"
    - "insert location explanation"
  not_default_as:
    - "Manual Commit Pack"
    - "Human Direct Commit execution file"
```

一文圧縮：

```text id="patch-only-final"
Patch-onlyはReview補助。
Manual Commit PackはFull Bodyがdefault。
```

### Ultra-Neutral Placeholder Railとの接続

Ultra-Neutral Placeholderで住所が開いた後は、AI側で本体payloadを深追いしない。

その時点で、GitHub上の編集面は開いている。

次にAIが出すべきものは、Patch-onlyではなく、原則としてFull Body Replacement Fileである。

```text id="placeholder-to-full-body"
TODO placeholderで住所を開く。
住所が開いたらAI側搬送を止める。
Full Body Replacement Fileを渡す。
人間が本来版を貼る。
```

English anchor:

```text id="placeholder-to-full-body-anchor"
Address opener first.
Full body handoff next.
```

### zip方針

Download付きManual Commit Packでも、zipは常に必要ではない。

```yaml id="zip-policy"
zip_policy:
  one_file: "zip不要"
  two_files: "基本zip不要"
  three_or_more_files: "zip検討"
  folder_structure: "zip推奨"
```

2ファイルまでは、個別Download linkの方がCopy & Pasteしやすい。

### Full Bodyの二種類

Full Bodyには二種類ある。

```yaml id="full-body-types"
full_body_types:
  exact_full_body:
    meaning: "既存本文をほぼ一字一句保持し、Patchだけ加える"
    use_when:
      - "既存本文を変えないことが要件"
      - "契約文・機械生成物・厳密保持が必要"
      - "ユーザーが差分だけを指定"

  successor_full_body:
    meaning: "原文を取得し、意味を保ち、改善した後継全文を作る"
    use_when:
      - "Guide改善"
      - "Ark-OKF化"
      - "重複整理"
      - "運用Rule追加"
      - "ユーザーが修正・改善を許可"
```

Guide文書では、ユーザーが修正・改善を許可している場合、Successor Full Body Replacementを使ってよい。

```text id="successor-full-body-core"
再構築できないのではない。
完全復元に固執しすぎない。
Guide改善なら、Full Body successorとして書き直せばよい。
```

### 例外

Full Bodyがdefaultである。

ただし、AIが完全な現行本文を安全に取得・再構成できない場合は、破壊的Full Bodyを作らない。

```yaml id="full-body-exception"
exception:
  when:
    - "complete current body cannot be safely reconstructed"
    - "tool output is truncated"
    - "existing SSOT integrity may be damaged"
    - "public/private boundary is unclear"
  do:
    - "do not invent full body"
    - "do not destructive overwrite"
    - "provide Patch Preview plus human edit instructions"
    - "ask for safer source or human upload if needed"
```

一文圧縮：

```text id="balanced-rule"
Full Bodyがdefault。
ただし、安全に全文を再構成できないなら、破壊的全文置換はしない。
```

## 6.9 Commit Message + Extended Description Pair

For important commits, AI should provide both a commit message and an extended description.

```text id="commit-pair"
Commit message:
  what changed

Extended description:
  why it changed
```

The commit message should be short, scope-aware, and future-readable.

The extended description should be one or two lines.

Use extended description when the reason matters for future AI or future human review.

```yaml id="extended-description-use"
extended_description_use_when:
  - "canonical guide update"
  - "SeedSkill / Skill file update"
  - "Human Direct Commit"
  - "Safety Block recovery"
  - "original-intended content replaces safety-softened content"
  - "future Living Review should understand why"
```

Example:

```text id="commit-pair-example"
Commit message:
docs(g_global): add full-body manual commit pack rule

Extended description:
Document that downloadable Manual Commit Packs should default to full-body replacement files, while patch-only downloads remain preview aids.
```

```yaml id="safe-options"
    use_when: "complete current body is safely available"

  successor_full_body_update:
    use_when: "user permits rewrite / improvement"
    result: "create improved full-body successor"

  staged_parent_integration:
    use_when: "companion has passed Reality Response"
    result: "later integrate short reference into parent guide"
```

## 8. Do / Do Not

Do:

```text id="do"
- Report the block honestly.
- Treat the exact internal safety reason as unknown.
- Separate confirmed tool result from inference.
- Preserve the original source as source_bootstrap.
- Test a smaller write unit.
- Use Patch Preview before retrying.
- Commit only after Human Seal.
- Protect existing canonical files from destructive overwrite.
- Use Full Body Manual Commit Pack when manual execution is the route.
- Use Successor Full Body when the user permits rewrite or improvement.
```

Do not:

```text id="do-not"
- Claim exact OpenAI internal safety reason.
- Assume the content is wrong.
- Weaken Root / Fruit Guard before diagnosis.
- Delete core structure before diagnosis.
- Repeatedly retry the same blocked payload.
- Treat this as a safety-check bypass technique.
- Run update_file without safe full replacement content.
- Provide patch-only download as the default Manual Commit Pack.
- Create destructive Full Body from incomplete source.
```
