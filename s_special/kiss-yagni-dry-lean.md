## 3. Four Principles

### 3.1 KISS — Keep It Simple

KISS means:

```text
Make the next faithful step simple enough to execute.
```

In Ark, KISS does not mean shallow.

It means:

```text
Deep structure, light next action.
```

Bad KISS:

```text
Oversimplify the truth.
Ignore necessary guardrails.
Remove meaning for the sake of speed.
```

Good KISS:

```text
Keep the visible action small.
Keep the hidden judgment clear.
Preserve the Root.
Remove avoidable friction.
```

Ark example:

```text
Instead of designing a full GitHub operating system first,
create one README,
verify Reality Response,
then continue.
```

### 3.2 YAGNI — You Aren’t Gonna Need It

YAGNI means:

```text
Do not build what Reality has not yet asked for.
```

In Ark, YAGNI protects the project from premature architecture.

Bad YAGNI:

```text
Refuse to prepare anything.
Use minimalism as avoidance.
Ignore likely near-term needs.
```

Good YAGNI:

```text
Reserve slots conceptually.
Create real files only when Reality Response requires them.
Avoid speculative folder sprawl.
```

Ark example:

```text
Mention future slots such as s_special/ or g_global/,
but do not create the folders until a real file needs to live there.
```

### 3.3 DRY — Don’t Repeat Yourself

DRY means more than “do not copy the same sentence twice.”

In Ark, DRY means:

```text
Do not maintain two co-equal sources of truth for one Living SSOT.
```

The deeper danger is not only content duplication.

The deeper danger is:

```text
canonical duplication
```

Japanese:

```text
正準性の二重化
```

This happens when two files, paths, or versions both appear to be the “real one.”

Example of a DRY failure:

```text
Local:
  S_kiss-yagni-dry-lean_v003.md

GitHub:
  s_special/kiss-yagni-dry-lean.md

Both are edited.
Both are treated as current.
Future AI does not know which one to trust.
```

Correct direction:

```text
GitHub canonical path:
  s_special/kiss-yagni-dry-lean.md

Local artifact:
  export / backup / private-depth / bootstrap only
```

DRY in Ark:

```text
One living source.
Many possible exports.
No co-equal duplicate canon.
```

### 3.4 Lean — Observe Flow, Patch Small

Lean means:

```text
Observe the flow.
Find friction.
Patch the smallest real bottleneck.
Check Reality Response.
```

In Ark, Lean is both a principle and a flow runtime.

It asks:

```text
Where did the flow actually slow down?
What did reality reveal?
What is the smallest patch that improves the next pass?
```

Bad Lean:

```text
Turn everything into efficiency.
Ignore faith, repentance, and human limits.
Optimize the fruit as if it were Root.
```

Good Lean:

```text
Let Reality Response expose friction.
Use KISS / YAGNI / DRY to reduce waste.
Move one faithful step.
Preserve Root Guard.
```

Ark formula:

```text
Reality Response drives Patch.
```

## 6. GitHub Canonical First as a DRY Case

The most important recent DRY case is GitHub Canonical First.

Before GitHub commit flow was proven, local Markdown files were useful:

```text
Preview
Download
Local save
Re-upload
Manual version management
```

That was appropriate during the discovery phase.

After the GitHub commit rail succeeded, the same safety pattern became unnecessary friction.

Old pattern:

```text
Local:
  G_chatgpt-github_v001.md

GitHub:
  g_global/chatgpt-github.md
```

New pattern:

```text
GitHub:
  g_global/chatgpt-github.md
```

The breakthrough:

```text
Markdown canonicality moved from local filename versioning to GitHub stable path + commit history.
```

Japanese:

```text
Markdownの正準性を、
ローカルvXXXファイル名から、
GitHub stable path + commit historyへ移した。
```

This is DRY in action.

It removes:

- dual Markdown
- version filename sprawl
- manual save friction
- re-upload loops
- uncertainty about latest version
- Future AI confusion

Rule:

```text
Public-safe Living Markdown is GitHub-canonical by default.
Local Markdown is exception/export/bootstrap only.
```

## 7. Canonical Duplication Guard

Canonical Duplication means:

```text
Two artifacts both appear to be the real source.
```

Japanese:

```text
正準性の二重化
```

This is more dangerous than simple copied text.

Copied text is annoying.

Duplicated canon is confusing.

It asks Future AI to answer impossible questions:

```text
Which file is real?
Which one is latest?
Which one should I update?
Which one should I cite?
Which one should the next thread inherit?
```

Canonical Duplication appears when:

```text
- local vXXX files keep evolving after GitHub canonical placement
- GitHub stable path and local versioned path are both treated as current
- README mentions one path but the active file lives elsewhere
- old handoff files remain operationally active without status change
- exports are mistaken for source
```

Guard:

```text
One canonical source.
Clear export status.
Stable GitHub path when public-safe.
Private/local only when necessary.
```

Status labels should be explicit:

```yaml
canonical: true
bootstrap_artifact: false
export_copy: false
private_depth: false
```

For old local files:

```yaml
status: "bootstrap_artifact_not_future_ssot"
canonical_ssot: "s_special/kiss-yagni-dry-lean.md"
```

## 8. Commit History as Version Ledger

GitHub canonical files should not carry version numbers in their filenames.

Bad GitHub canonical pattern:

```text
s_special/S_kiss-yagni-dry-lean_v001.md
s_special/S_kiss-yagni-dry-lean_v002.md
s_special/S_kiss-yagni-dry-lean_v003.md
```

Good GitHub canonical pattern:

```text
s_special/kiss-yagni-dry-lean.md
```

Versioning belongs in:

```text
frontmatter
commit history
tags or snapshots only when truly needed
```

Core rule:

```text
Stable path is the address.
Commit history is the version ledger.
```

Japanese:

```text
Pathは住所。
Git履歴はVersion台帳。
```

This improves:

- Future AI lookup
- human navigation
- update simplicity
- diff review
- DRY
- long-term maintainability

## 9. Markdown / Artifact Operations

Use this lens whenever creating, updating, or placing Markdown artifacts.

Decision checklist:

```text
1. Is this public-safe?
2. Is this intended to become a Living SSOT?
3. Does a canonical GitHub path already exist?
4. Would a local duplicate create canonical confusion?
5. Is a local export explicitly needed?
6. What is the smallest faithful next step?
```

Default for public-safe Living Markdown:

```text
Plan Mode
-> Preview
-> Human Seal
-> GitHub Commit
-> Reality Review
```

Local Markdown creation is not default.

Local artifacts are allowed when:

```text
- private-depth content must remain outside public GitHub
- user explicitly asks for downloadable file
- GitHub connector is unavailable
- temporary draft is needed before canonical placement
- output is PDF, DOCX, image, ZIP, spreadsheet, or other non-canonical artifact
```

Do not make local and GitHub Markdown co-equal.

Do not create “just in case” exports unless the user asks or Reality Response requires it.

## 11. Failure Modes

### 11.1 KISS failure

```text
The system becomes too elaborate to execute.
```

Patch:

```text
Return to the smallest visible next step.
```

### 11.2 YAGNI failure

```text
Folders, files, modes, or schemas are created before Reality asks for them.
```

Patch:

```text
Reserve conceptually.
Do not create physically yet.
```

### 11.3 DRY failure

```text
Two files become co-equal sources of truth.
```

Patch:

```text
Choose one canonical path.
Mark the other as export/bootstrap/archive.
```

### 11.4 Lean failure

```text
The team optimizes a theory instead of observing actual flow.
```

Patch:

```text
Ask: what actually happened?
```

### 11.5 FullRail failure

```text
Semi-automation becomes fake autopilot.
```

Patch:

```text
Restore Human Seal.
```

### 11.6 GitHub Canonical failure

```text
Public-safe and private-depth materials are mixed.
```

Patch:

```text
Public-safe to GitHub.
Private-depth stays private/local.
```
