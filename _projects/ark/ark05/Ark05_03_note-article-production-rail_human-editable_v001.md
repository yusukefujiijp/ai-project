## 1. Purpose

Article Production Railとは、AIと人間が協働して、note有料記事を **週1本・500円販売・内部980円品質候補** へ近づけるための制作Workflowである。

このRailの目的は、AIに記事を丸投げすることではない。  
このRailの目的は、記事を急いで量産することでもない。

目的は、各Layerを正しい順序で通し、記事Draftを生きた試験体として鍛え、販売可能な形へ整え、次回TemplateへHarvestすることである。

## 3. Core Operating Spirit

### 3.1 Deep Breath Step-by-Step

Ark05では、焦らない。

```text
Deep Breath Step-by-Step:
  1. 今扱うLayerを確認する
  2. そのLayerの目的を一文で言う
  3. 小さくDraftまたはBlockを作る
  4. Reviewを受ける
  5. 違和感を言語化する
  6. 必要な修正だけ行う
  7. 次Layerへ進む
```

姿勢:

```text
焦らず、しかし止まらず。

発想は自由に、検査は厳密に。
歩みはゆっくり、方向はReality Gateへ。
```

### 3.2 Layered Overcapture Guard

初期設計・初期Draftでは、不足より過剰を優先する。

```text
書き過ぎ:
  後でCut / Split / Extractできる

書き漏れ:
  後で発見・補完するのが難しい
```

ただし、過剰捕獲には条件がある。

```text
Overcapture OK:
  - Layer構造がある
  - Block単位で編集可能
  - Version情報がある
  - 後からCut / Split / Extractできる
  - 死んだ羅列ではなくLiving Boardになっている
```

問題は長さではない。  
問題は、Layerを失った長文化である。

Move37は奇抜さではない。

```text

- X投稿へ分解できるか
```

## 2.4 Gate Stop Rule


# Rail 3: Draft Generation Gate

## 3.1 Purpose

Draft Generation Gateでは、AIと協働して初稿を作る。

ここでの目的は、完成記事を書くことではない。  
目的は、Review可能な素材を出すことである。

```text
Draft目的:
  完成ではなく、検査可能な素材を出す。
```

## 3.2 Draft Generation Principle

```text
不足より過剰。
ただしBlock構造で書く。
```

初稿では、書き漏れを恐れる。  
長くなってもよい。  
ただし、後でCut / Split / ExtractできるようにBlockで書く。

## 3.3 Draft Blocks

```text
- Title Candidate
- Opening Hook
- Reader Pain
- Reality Gate
- Core Thesis
- Method
- Example / Analogy
- AI-Human Collaboration
- Template / CheckList
- Experiment Log
- Evidence
- Next Action
- Closing
```

すべて必須ではない。  
ただし、980円品質候補を狙うなら、Template / Evidence / 実験ログ / Next Action のどれかは必要である。

## 3.4 AI Draft Role

AIに求めること:

```text
- 構成案を出す
- Draftを書き出す
- 読者Painを補う
- 一般論化しそうな箇所を警告する
- Template候補を出す
- X展開しやすいKernelを残す
```

AIに求めないこと:

```text
- 成功者として語らせる
- 売上保証させる
- 人間の実感を捏造させる
- 最終判断を任せる
```

## 3.5 Gate Stop Rule

止まる条件:

```text
- Draftが一般論だけ
- Human実感がない
- Reader Painが消えた
- 成功者ぶりが出た
- AI丸投げ感が強い
```

戻るGate:

```text
Article Kernel Gate または Theme / Reader Pain Gateへ戻る。
```

# Rail 5: Structural Repair Gate

## 5.1 Purpose

Structural Repair Gateでは、Draft Triageで見つかった最大の詰まりを直す。

ここでは、全部直そうとしない。  
最初の一手に集中する。

## 5.2 Repair Targets

```text
- Reader Pain
- Article Kernel
- 構成
- Buyer Value
- Template不足
- 実験ログ不足
- Evidence不足
- Integrity違反
- AIっぽい一般論
```

## 5.3 Repair Rule

```text
One Repair at a Time:
  まず最大の詰まりを1つ直す。
```

理由:

```text
全部直そうとすると、記事の生命力が消える。
最大の詰まりを直すと、他の問題も自然に動くことがある。
```

## 5.4 Repair Output

```text
- Revised Kernel
- Revised Outline
- Added Template
- Added Experiment Log
- Added Evidence Candidate
- Rewritten Opening
- Integrity Correction
```

## 5.5 Gate Stop Rule

止まる条件:

```text
- 修正してもBuyer Valueが立たない
- Kernelが何度も崩れる
- Theme自体が弱い
```

戻るGate:

```text
Theme / Reader Pain Gateへ戻る。
```

# Rail 7: Evidence / Template Reinforcement Gate

## 7.1 Purpose

Evidence / Template Reinforcement Gateでは、Full Critical Reviewで不足した要素を補強する。

目的:

```text
記事を500円販売品質、可能なら980円品質候補へ近づける。
```

## 7.2 Reinforcement Types

```text
External Evidence:
  研究 / 公開データ / 公式情報 / 信頼できる記事 / 事例

Internal Evidence:
  自分の実験ログ / 制作過程 / 失敗 / 観察 / AIとの対話

Structural Evidence:
  Template / CheckList / Before-After / 判断表 / Prompt / Workflow
```

## 7.3 Evidence Enough Rule

Evidenceは多ければよいわけではない。

```text
1記事につき、強いEvidenceは1〜3個でEnough。
ただし、実験ログやTemplateは積極的に入れる。
```

Evidence過剰になると、記事が重くなる。  
Evidence不足だと、記事が浮く。

## 7.4 Template Reinforcement

980円品質候補を狙うなら、Templateが強い。

Template例:

```text
- 読者用CheckList
- 判断表
- AI Prompt
- Before / After表
- 5-step Workflow
- 自分用記録シート
```

Templateは、読者が記事を保存する理由になる。

## 7.5 Gate Stop Rule

止まる条件:

```text
- Evidence探しで出荷が止まる
- Templateが複雑すぎる
- 記事が調査レポート化する
```

戻るGate:

```text
Full Critical Review Gate または Sales Compression Gateへ進む。
```

# Rail 8: 500円 Sales Compression Gate

## 8.1 Purpose

500円 Sales Compression Gateでは、980円品質候補の素材を、500円販売記事として読める形へ整える。

Compressionは価値削除ではない。  
Compressionとは、読者が価値を受け取りやすい形へ整えることである。

## 8.2 Compression Targets

削る候補:

```text
- 長すぎる前置き
- 重複説明
- AIっぽい一般論
- 読者価値に直結しない枝葉
- 誇大表現
- 説明しすぎた内部事情
```

残すもの:

```text
- Article Kernel
- Reader Pain
- Buyer Value
- Template
- 実験ログ
- Evidence
- Next Action
- Human誠実性
```

## 8.3 Compression Rule

```text
Expand → Structure → Compress → Ship
```

順番が重要である。

```text
Expand:
  不足より過剰で素材を捕獲する

Structure:
  Layer / Blockへ分ける

Compress:
  読者価値に沿って圧縮する

Ship:
  Human Final Seal後に出荷する
```

## 8.4 500円と980円の関係

```text
External:
  500円販売

Internal:
  980円品質候補
```

500円だから薄くしない。  
500円だからこそ、読者が「思ったより濃い」と感じる余剰価値を作る。

## 8.5 Gate Stop Rule

止まる条件:

```text
- CompressionでTemplateやEvidenceが消えた
- 読者価値ではなく短さを優先した
- 誠実性が薄くなった
- Kernelが消えた
```

戻るGate:

```text
Evidence / Template Reinforcement Gate または Structural Repair Gateへ戻る。
```

```text
- 読者が保存したくなるか

```

## 9.4 Move37 Scale Slot

# Rail 10: Harvest Gate

## 10.1 Purpose

Harvest Gateでは、今回の記事制作から次回以降に残すべきものを抽出する。

記事は一回で終わらない。  
記事制作そのものを次回Systemへ還元する。

## 10.2 Harvest Targets

```text
- 良かったPrompt
- 効いたReview
- 足りなかったEvidence
- 次回使えるTemplate
- X投稿Kernel
- PDF章候補
- 読者Pain
- Buyer Valueの発見
- Integrity Guardの改善
- Ark05 Core Patch候補
```

## 10.3 Harvest Format

```text
Keep:
  次回も残すもの

Change:
  次回変えるもの

Add:
  次回追加するもの

Remove:
  次回削るもの

Patch:
  Ark05 Core 3へ反映する候補

Next:
  次の記事制作で最初に試すこと
```

## 10.4 Harvest Rule

```text
Harvestは反省文ではない。
Harvestは次回の制作速度と品質を上げる資産化である。
```

## 10.5 Gate Stop Rule

止まる条件:

```text
- Harvestが感想だけ
- 次回に残るActionがない
- Ark05 Coreへ戻るPatch候補がない
```

戻るGate:

```text
Review結果を再確認し、Keep / Change / Add / Remove / Patchへ分ける。
```
